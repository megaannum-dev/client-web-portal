## Read this first — why this is phased, not fully parallel

Unlike refactor 001 (a pure file move), 002 has a real dependency graph: the ORM model layer is foundational, the Alembic baseline must be taken **before** the models change, and the auth dependency chain (`deps → actions + repositories → service → router`) is inherently ordered. So "execute each step in parallel" is honoured **within each phase**, but phases are sequential. The parent agent must not collapse them.

```
Phase 0  Preflight (parent only) — branch, DB snapshot, server health
   │
Phase 1  Alembic bootstrap + BASELINE current schema      (Agent 1)        ← must precede model edits
   │
Phase 2  Data model: enums, columns, profiles, role prop  (Agent 2)        ← foundational, blocks all code
   │
   ├───────────────┬───────────────┬───────────────┐
Phase 3A claims    3B actions       3C repositories  (parallel — Agents 3A/3B/3C)
   │               │                │
   └───────────────┴───────────────┘
   │
Phase 4  Auth dependency split + require_action rework    (Agent 4)
   │
Phase 5  Auth service + auth router + provisioning        (Agent 5)
   │
Phase 6  Documents-router portal gate edit                (Agent 6)
   │
Phase 7  Wiring / main.py sanity                          (Agent 7)
   │
Phase 8  Author migration revision + seeding package      (Agent 8)        ← authored after Phase 2; RUN in V3
   │
Phase 9  Docker migration wiring (Dockerfile + entrypoint) (Agent 9)
   │
Validation (parallel):  V1 static  ·  V2 logic/module tests  ·  V3 migration+seed test
   │
   └──► V4  Full-stack Docker rebuild + integration test    (parent)        ← only after V1/V2/V3 green
```

Phase 3A (claims) technically only needs `Settings`, but it is scheduled after Phase 2 to keep one clean barrier. 3A/3B/3C are genuinely independent and run concurrently.

| **E-3** | §H.0 `User` relationships | `client_profiles` has **two** FKs into `users` (`user_id → users.id`, `assigned_rm_uid → users.firebase_uid`). The `User.client_profile` relationship is therefore **ambiguous** and SQLAlchemy will error at mapper configuration without disambiguation. | The explicit `primaryjoin` in §H.0 is **mandatory**, and additionally pass `foreign_keys="ClientProfile.user_id"` on `User.client_profile`. `admin_profile` has a single FK and needs only the `primaryjoin`. |
| **E-4** | §H.0 `role` property + §H.1 `UserOut` | A Python `@property` named `role` feeding Pydantic `from_attributes` is correct at runtime but mypy may flag the ORM attribute shape. | Type the property `-> str`. If mypy objects to the property/column name overlap, add a targeted `# type: ignore[...]` on the property line only — do **not** silence file-wide. |
| **E-5** | §A.3 vs §B | If the baseline revision is autogenerated **after** the new models exist, it will wrongly include the new tables. | Order is fixed by the phase plan: **Agent 1 baselines the CURRENT schema and runs `alembic upgrade head` before Agent 2 touches any model.** Do not reorder. |
| **E-6** | §C.3 migration | `op.add_column("users", "portal")` then backfill then `alter_column(... nullable=False)`. On MySQL/MariaDB, `alter_column` to NOT NULL requires the existing type restated. | Keep `existing_type=sa.String(16)` on the `alter_column` (already in §C.3). Verify the dialect accepts the enum-as-string; we store portal as `VARCHAR(16)` (`native_enum=False`), so a plain `String` is correct — do not emit a native ENUM. |

## Phase 9 — Agent 9: Docker migration wiring

**Prereq:** Phase 1 (Alembic exists) + Phase 8 (revision authored). **Why this is needed:** the current `api-backend/Dockerfile` copies only `app/` and runs `uvicorn` directly — so `alembic/`, `alembic.ini`, and `scripts/` are absent from the image and **no migration runs on startup**. Without this phase the rebuilt container boots against an unmigrated schema.

### File edits

**`api-backend/Dockerfile`** — copy the migration assets and run migrations before the server. Add after `COPY app ./app`:
```dockerfile
COPY alembic ./alembic
COPY alembic.ini ./alembic.ini
COPY scripts ./scripts
```
Replace the final `CMD` with an entrypoint that migrates then serves:
```dockerfile
CMD ["sh", "-c", "alembic upgrade head && uvicorn app.main:app --host 0.0.0.0 --port 8000"]
```

Notes:
- `requirements-dev.txt` (pytest/httpx, D-B) is **not** installed in the image — it is host/CI-only. Production image installs `requirements.txt` only (which now includes `alembic` via Agent 1).
- `Base.metadata.create_all()` in the lifespan is now redundant (Alembic owns DDL). Leave it (harmless on a migrated DB) **or** remove it — but if removed, the entrypoint `alembic upgrade head` becomes the sole schema authority, which is the cleaner end state. Record the choice.
- The seeding script (`scripts/seed_profiles/seed.py`) is **not** auto-run by the entrypoint — it is a manual, dev-only, one-off (it fabricates dummy data). It is copied into the image only so it can be invoked manually (`docker compose exec api-backend python -m scripts.seed_profiles.seed`) against a disposable DB.

### Verify
`docker build -f api-backend/Dockerfile api-backend/` succeeds (image builds, assets present). Do not run the stack here — that is V4.

## Resolved Decisions (settled 2026-06-10 — baked into the phases above)

- **D-A — Disposable database.** Agents target a **throwaway DB only**, never the real ~18-row source. Phase 0 stands up a local MariaDB/MySQL, loads the Phase-0 dump (or the synthetic fixture for V3), and exports `DATABASE_URL` pointing at it for all agent and validation work. The real DB is never opened by any agent.
- **D-B — Test dependencies added.** Create `api-backend/requirements-dev.txt` with `pytest`, `httpx` (for FastAPI `TestClient`), and `pytest` plugins as needed. V2/V3 install from it. This is a prerequisite, not optional.
- **D-C — V2 on in-memory SQLite; V3 on real MySQL/MariaDB.** Module/logic tests use SQLite (models are portable). The migration test stays on a real MySQL/MariaDB because `alter_column ... NOT NULL` and the DDL are dialect-sensitive.
- **D-D — Claim logic tested with a mocked, disposable `firebase_admin.auth`.** V2 asserts the claim-setting *logic* by patching `firebase_admin.auth` (test-only, disposable — never touches a real Firebase project). Live claim propagation is verified manually later (Proposal §11 Q2 / FR-2).
- **D-E — No `/api/client` · `/api/admin` scaffold (my best-judgement call).** Empty prefixes are not introduced — they would be dead routers and FastAPI mounts nothing useful for them. The trees are added when the first real client/admin domain endpoint exists, matching Proposal §7.1 and avoiding 404-shaped clutter in the OpenAPI schema. Phase 7 explicitly skips this.
- **Parallelism note (unchanged):** genuine concurrency is limited to Phase 3 (3A/3B/3C) and the three validation agents; the rest is dependency-ordered to avoid races on `models/users.py` and the auth import chain.
